# API Key Leak Scanner

Tägentliche, intensive Suche nach exponierten API-Keys in GitHub-Repositories und Gists,
insbesondere in Projekten von Personen mit geringer Code-Erfahrung, die AI-Code-Generatoren
(ChatGPT, Claude, Copilot) verwenden und dabei API-Keys unbewusst offenlegen.

## Übersicht

Der Scanner nutzt die **GitHub Search API** in drei Phasen:

1. **AI-Generated Repos** — Sucht nach Repos mit typischen AI-Indikatoren in Readmes/Beschreibung (`generated by ChatGPT`, `AI-generated`, `built with Claude`, usw.) und scannt deren `.env`, `config.json`, `secrets.json`, etc.
2. **Direct Key Search** — Sucht direkt in Code nach typischen API-Key-Patterns in `.env`- und Config-Dateien unter diesem Suchwort mit `path:.env` oder `path:config` Filtern. Prüft ob gefundene Repos AI-generiert sind.
3. **Gists Scan** — Durchsucht öffentliche Gists nach exponierten Schlüsseln.

## Was wird gefunden

Der Scanner erkennt **18+ Key-Typen**, darunter:

| Typ | Muster |
|-----|--------|
| OpenAI API | `sk-`, `sk-proj-` |
| Anthropic / Claude | `sk-ant-` |
| Google Gemini | `AIza...` (35 Zeichen) |
| GitHub PAT / OAuth | `ghp_`, `ghs_`, `gho_`, `ghr_` |
| Stripe | `sk_live_`, `sk_test_`, `pk_live_` |
| AWS Access Key | `AKIA` + 16 Zeichen |
| Azure | `*_key = "..."` |
| Slack / Discord / Telegram Bots | Bot-Token-Muster |
| xAI / Grok | `xai_`, `grok_` Prefix |
| DeepSeek | `deepseek_`, `ds_` |
| JWT, Generic API Keys | weitere Muster |

## Was NICHT gefunden / NICHT geprüft wird

- **Balance/Guthaben** wird **nicht** automatisch abgefragt. Es würde echte API-Calls verursachen.
  Es liegt in der Verantwortung des Benutzers, gefundene Keys auf ihre Gültigkeit zu prüfen.
  Der Scanner prüft nur Format, Länge, Präfix und Filterung von offensichtlichen Placeholdern.
- **Private / nicht-öffentliche Repos** — nur öffentliche GitHub-Suche
- **Komplexe verschleierte Keys** — keine Base64-, Hex- oder verschlüsselten Varianten
- **Schlüssel in Binärdateien** — keine Bytecode- oder kompilierten Artefakte

## Arbeitsweise pro Phase

1. GitHub API Code Search (ca. 200–500 Calls pro Lauf)
2. Filterung auf verdächtige Dateinamen (`.env`, `config.*`, `secrets.*`, etc.)
3. Für Treffer: Datei-Inhalt holen, Regex-Matching auf Key-Patterns
4. Deduplizierung (doppelte Keys werden nur einmal gezählt)
5. Repo-Beschreibung + README auf AI-Indikatoren prüfen

## Rate Limits

| Authentifiziert | Nicht auth. |
|-----------------|-------------|
| 5000 req/h Code-Suche | 10 req/min Code-Suche |
| 60 req/h Repo-Suche | 30 req/h Repo-Suche |

Ohne GitHub-Token ist die Suche stark eingeschränkt. Setze `GITHUB_TOKEN` oder `GH_TOKEN`
in der Umgebung für produktiven Betrieb:

```bash
# In ~/.bashrc oder ~/.bash_profile
export GITHUB_TOKEN="ghp_..."   # Nur mit repo-Scope (kein Admin!)
```

## Ausführung & Output

Der Scanner läuft im Terminal, gibt eine ASCII-Box mit Statistik + Top-20-Treffern aus
und schreibt ein Detail-JSON nach `~/.hermes/data/key-scanner/scan_YYYYMMDD_HHMMSS.json`.

## Wichtig
Enthält alle Treffer mit Typ, Preview, Dateipfad, Repo-Link und `ai_generated`-Flag.

## Cron-Job (täglich um 20 Uhr)

Der Skill wird via Cron-Job automatisiert. Der Cronjob:
- Startet täglich um 20:00 Uhr
- Führt den Scanner mit 45 Min. Laufzeit aus
- Liefert den Report als Nachricht zurück

## Konfiguration via Environment-Variablen

| Variable | Standard | Beschreibung |
|----------|---------|--------------|
| `GITHUB_TOKEN` / `GH_TOKEN` | — | GitHub Token für höhere Limits |
| `SCAN_DURATION` | `45` | Dauer in Minuten |
| `RATE_LIMIT_PAUSE` | `0.8` | Pause zwischen API-Calls (Sekunden) |
| `OUTPUT_DIR` | `~/.hermes/data/key-scanner` | JSON-Output-Pfad |

## Unterstützende Dateien

- `templates/cron-wrapper.sh` — Wrapper-Skript für Cron-Jobs mit Logging, Timeout und Fehlerbehandlung
- `references/cron-setup-20260517.md` — Session-spezifische Cron-Einrichtung und Parameter aus der ersten produktiven Test-Session (2026-05-17)

## Ethische Grundsätze

Dieser Scanner dient zur **eigenen Überwachung** (Security-Monitoring) und zum Finden
offengelegter Schlüssel zur **Responsible Disclosure**. Gefundene Keys sollten nur zur
Verifizierung und anschließenden Meldung an die Betreiber verwendet werden — **nicht**
für unbefugten Zugriff.
